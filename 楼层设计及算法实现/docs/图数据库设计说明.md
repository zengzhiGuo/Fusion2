# 室内导航图数据库设计说明

## 概述

本系统采用关系型数据库实现图结构，用于室内导航路径规划。图由节点（Nodes）和边（Edges）组成。

## 图的组成

### 节点（Nodes）

节点存储在两张表中：

#### 1. doors 表（房间/楼道/出入口节点）
- 字段：id, room_name, room_type, map_id, x, y
- room_type:
  - 1: 房间
  - 2: 楼道
  - 3: 出入口
- 坐标：(x, y) 表示节点在地图上的位置

#### 2. stairways 表（楼梯节点）
- 字段：stairway_id, stairway_name, map_id, upper_map_id, lower_map_id, x, y
- 特殊属性：连接不同楼层的节点

### 边（Edges）

边存储在 graph_edges 表中：

#### graph_edges 表结构
```sql
- id: 主键
- edge_id: 边的唯一标识
- from_node_type: 起点节点类型（1-doors, 2-stairways）
- from_node_id: 起点节点ID
- to_node_type: 终点节点类型（1-doors, 2-stairways）
- to_node_id: 终点节点ID
- weight: 边的权重（距离，单位：米）
- is_bidirectional: 是否双向（1-双向，0-单向）
- map_id: 所在楼层ID
- is_active: 是否启用
- is_accessible: 是否可通行
```

## 图的类型

### 1. 双向图（Undirected Graph）

大多数室内通道是双向的，可以双向通行。

实现方式：
- 设置 `is_bidirectional = 1`
- 只需存储一条边记录
- 算法中自动处理双向关系

示例：
```sql
-- 房间A ↔ 走廊B（双向）
INSERT INTO graph_edges 
(edge_id, from_node_type, from_node_id, to_node_type, to_node_id, 
 weight, is_bidirectional, map_id)
VALUES 
('edge_1', 1, 'room_a', 1, 'corridor_b', 5.0, 1, 'floor3');
```

### 2. 单向图（Directed Graph）

某些特殊通道只能单向通行（如紧急出口、单行道）。

实现方式：
- 设置 `is_bidirectional = 0`
- 只能从 from_node 到 to_node

示例：
```sql
-- 紧急出口 → 室外（单向）
INSERT INTO graph_edges 
(edge_id, from_node_type, from_node_id, to_node_type, to_node_id, 
 weight, is_bidirectional, map_id)
VALUES 
('edge_2', 1, 'emergency_exit', 1, 'outdoor', 3.0, 0, 'floor1');
```

## 权重计算

### 自动计算距离

可以根据节点坐标自动计算欧几里得距离：

```sql
-- 计算两点之间的距离
SELECT SQRT(
    POW(node2.x - node1.x, 2) + 
    POW(node2.y - node1.y, 2)
) AS distance
FROM doors node1, doors node2
WHERE node1.id = 'room_a' AND node2.id = 'room_b';
```

### 手动设置权重

也可以手动设置权重，考虑其他因素：
- 实际步行距离（考虑障碍物）
- 通行时间
- 拥挤程度
- 优先级（如无障碍通道）

## 图的查询

### 1. 查询节点的所有邻居

```sql
-- 查询节点的所有邻接节点（考虑双向边）
SELECT 
    CASE 
        WHEN from_node_id = 'node_1' THEN to_node_id
        ELSE from_node_id
    END AS neighbor_id,
    CASE 
        WHEN from_node_id = 'node_1' THEN to_node_type
        ELSE from_node_type
    END AS neighbor_type,
    weight
FROM graph_edges
WHERE (
    (from_node_type = 1 AND from_node_id = 'node_1')
    OR (to_node_type = 1 AND to_node_id = 'node_1' AND is_bidirectional = 1)
)
AND is_active = 1 
AND is_accessible = 1;
```

### 2. 查询楼层内的所有边

```sql
SELECT * FROM graph_edges
WHERE map_id = 'floor3'
AND is_active = 1;
```

### 3. 查询跨楼层的边（楼梯连接）

```sql
SELECT e.*, s.upper_map_id, s.lower_map_id
FROM graph_edges e
JOIN stairways s ON (
    (e.from_node_type = 2 AND e.from_node_id = s.stairway_id)
    OR (e.to_node_type = 2 AND e.to_node_id = s.stairway_id)
)
WHERE e.is_active = 1;
```

## 路径规划算法

### 支持的算法

1. **Dijkstra算法**（最短路径）
   - 适用于有权重的图
   - 找到从起点到终点的最短路径

2. **A*算法**（启发式搜索）
   - 使用启发函数加速搜索
   - 适合大规模图

3. **BFS算法**（广度优先搜索）
   - 适用于无权重或权重相同的图
   - 找到最少跳数的路径

### 算法实现要点

1. **构建邻接表**
   ```java
   Map<String, List<Edge>> adjacencyList = new HashMap<>();
   // 从 graph_edges 表加载数据
   ```

2. **处理双向边**
   ```java
   if (edge.isBidirectional()) {
       adjacencyList.get(fromNode).add(new Edge(toNode, weight));
       adjacencyList.get(toNode).add(new Edge(fromNode, weight));
   }
   ```

3. **跨楼层处理**
   - 楼梯节点连接不同楼层
   - 需要在路径中标记楼层切换

## 图的维护

### 1. 自动创建边

当添加新节点时，可以自动创建与附近节点的边：

```java
// 伪代码
void autoCreateEdges(Node newNode) {
    List<Node> nearbyNodes = findNodesWithinRadius(newNode, radius);
    for (Node neighbor : nearbyNodes) {
        double distance = calculateDistance(newNode, neighbor);
        createEdge(newNode, neighbor, distance, true);
    }
}
```

### 2. 动态更新边

通过 `is_accessible` 字段动态控制通道：

```sql
-- 临时关闭某条通道
UPDATE graph_edges 
SET is_accessible = 0 
WHERE edge_id = 'edge_123';

-- 重新开放通道
UPDATE graph_edges 
SET is_accessible = 1 
WHERE edge_id = 'edge_123';
```

### 3. 删除节点时的处理

删除节点时，需要同时删除相关的边：

```sql
-- 删除与节点相关的所有边
DELETE FROM graph_edges
WHERE (from_node_type = 1 AND from_node_id = 'node_to_delete')
   OR (to_node_type = 1 AND to_node_id = 'node_to_delete');
```

## 优化建议

### 1. 索引优化

已创建的索引：
- `idx_from_node (from_node_type, from_node_id)`
- `idx_to_node (to_node_type, to_node_id)`
- `idx_map_id (map_id)`

### 2. 缓存策略

- 将图结构加载到内存中
- 使用 Redis 缓存常用路径
- 定期更新缓存

### 3. 分层存储

- 按楼层分区存储边
- 减少跨楼层查询的开销

## 扩展功能

### 1. 多种路径偏好

- 最短距离路径
- 最少楼层切换路径
- 无障碍路径（避开楼梯）
- 最少拥挤路径

### 2. 实时路径调整

- 根据人流量动态调整权重
- 避开临时封闭的通道
- 考虑电梯等待时间

### 3. 路径可视化

- 在前端地图上绘制路径
- 显示路径长度和预计时间
- 提供分步导航指引

## 示例场景

### 场景1：从房间A到房间B（同楼层）

```
房间A → 走廊1 → 走廊2 → 房间B
```

涉及的边：
1. edge_1: 房间A → 走廊1 (双向)
2. edge_2: 走廊1 → 走廊2 (双向)
3. edge_3: 走廊2 → 房间B (双向)

### 场景2：从3层到1层（跨楼层）

```
3层房间 → 3层走廊 → 楼梯(3层) → 楼梯(1层) → 1层走廊 → 1层出入口
```

涉及的边：
1. edge_4: 3层房间 → 3层走廊 (双向)
2. edge_5: 3层走廊 → 楼梯 (双向)
3. edge_6: 楼梯(3层) → 楼梯(1层) (双向，跨楼层)
4. edge_7: 楼梯 → 1层走廊 (双向)
5. edge_8: 1层走廊 → 1层出入口 (双向)

### 场景3：紧急疏散（单向路径）

```
房间 → 紧急出口 → 室外（单向）
```

涉及的边：
1. edge_9: 房间 → 紧急出口 (双向)
2. edge_10: 紧急出口 → 室外 (单向)

## 总结

通过 `graph_edges` 表实现的图结构具有以下优势：

1. **灵活性**：支持双向和单向边
2. **可扩展性**：易于添加新节点和边
3. **动态性**：可以实时更新边的状态
4. **高效性**：通过索引优化查询性能
5. **通用性**：适用于各种路径规划算法

这种设计为室内导航系统提供了坚实的数据基础。
